00 — Overview

What This System Is

Purpose
A hosted web app that manages all AI film projects — client and personal — through a structured, phase-locked production pipeline from raw script to finished storyboard.
Primary User
Khalil (director/producer), optionally shared with clients for review and approval at each phase gate.
Core Concept
Script in → everything extracted automatically → approve phase by phase → storyboard out. Nothing moves forward without a lock.
Hosting Target
Vercel (frontend) + Supabase (database + auth) + Vercel Edge Functions (API layer). Google Imagen for all image generation.
01 — Production Pipeline

Phase-by-Phase Breakdown

01

Project Creation & Asset Ingestion

User creates a new project (client or personal), uploads the script, treatment, director notes, or any combination of documents. System accepts PDF, DOCX, and plain text. Files are stored and associated with the project record.

File Upload (PDF, DOCX, TXT) Project Type: Client / Personal Multi-doc support
UI Layer
02

LLM Extraction Engine

Uploaded documents are passed to Claude (via Anthropic API). The LLM extracts and structures: all characters (with physical descriptions, personality, role), all scenes (location, time of day, action, mood), props per scene, wardrobe per character per scene, episode/act structure, and thematic notes. Output is stored as structured JSON in the database.

Anthropic Claude API Characters + Descriptions Scene Breakdown Props + Wardrobe Episode Structure JSON Output → DB
API Core
03

Film Bible (Auto-Generated)

Extracted data is rendered into a beautiful, web-native Film Bible. Think: a polished editorial document inside the app — not a PDF export. Sections for characters, locations, tone/mood, episode breakdown. Khalil or the client can read, annotate, and request edits before anything is locked. Bible is version-controlled.

Web-rendered document Annotation / Edit requests Phase Gate: Must approve to proceed Version history
UI Layer
04

AI Casting (Character Visualization)

For each character, Google Imagen generates 10 visual variations using the extracted physical description as the prompt base. Khalil reviews each set: approve, reject with notes ("too old", "different skin tone"), or flag for revision. Approved characters move to the pose phase.

Google Imagen API 10 variations per character Approve / Reject / Note Prompt built from extraction data
API Core
05

Character Lock & Reference Poses

Approved cast headshot is used as the seed for a reference pose shoot — front, 3/4, profile — generated via Imagen. These locked reference images become the character's canonical visual identity, used in all downstream storyboard generation. Locked images are stored in the Film Bible.

Google Imagen API Front / 3-quarter / Profile Locked into Film Bible Phase Gate: All characters must lock
API Core
06

Location & Scene Bible

Same approval loop as casting, but for locations. Imagen generates visual reference images for each location based on extracted scene descriptions. Approved locations lock into the Scene Bible. Scene-level details (time of day, mood, props in frame) are editable before lock.

Google Imagen API Per-location visual references Scene-level metadata Phase Gate: All locations must lock
API Core
07

Storyboard Generation

With all assets locked, Khalil selects a scene to storyboard. The system composes prompts for each shot using: the locked character reference images, the locked location image, the scene action and mood from the extraction, and the shot description. Imagen generates a panel-by-panel storyboard. Panels can be individually regenerated or annotated with shot notes (camera angle, lens, movement).

Google Imagen API Character + Location compositing Shot-by-shot panels Camera / Lens / Movement notes Per-panel regen Exportable
Output

05 — Open Questions

Decisions Still Needed

Q1
Client access model — Do clients get their own login, or does Khalil share a read-only link per project? (Affects auth complexity significantly.)
Option A: Full client auth (Clerk/Supabase) · Option B: Shareable magic link per project (simpler, faster to build)
Q2
Google Imagen access — Imagen 3 is currently available via Google Vertex AI, which requires a GCP project and billing setup. Do you have a GCP account, or do we need to plan that setup step?
Alternative fallback: fal.ai (Flux) until Imagen is configured — same API pattern, easy swap.
Q3
Character consistency across storyboard panels — Imagen doesn't natively lock character identity across generations. What's the acceptable approach?
Option A: Use approved headshot as image reference in each prompt · Option B: Accept variation, handle consistency in post (Kling/Veo) · Option C: Explore ControlNet-style seeds via a different provider
Q4
Export format — What does the final storyboard output look like? Web-view only, PDF export, or slide deck (Google Slides / PPTX)?
This affects whether we need a PDF generation layer (e.g. Puppeteer or a third-party export service).
Q5
Build approach — Claude Code (autonomous, terminal-based) vs. Claude + Cursor vs. traditional dev? Given your GitHub familiarity, Claude Code may be the fastest path.
This determines how we structure the first build sprint and what kind of scaffolding docs we prep next.
